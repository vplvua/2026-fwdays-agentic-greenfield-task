# План агентних циклів — Сервіс-деск Mini (POC)

> **Версія:** 1.1 · **Створено:** 2026-07-06 · **Оновлено:** 2026-07-07
> **Спирається на:** [PRD](./PRD.md) (v1.0) · [реєстр ADR](./adr/README.md)
>
> POC закривається за **7 циклів** (Е-1…Е-7; допустимо злиття до 5 — див. §3).
> **Цикл** = один агентний прохід: постановка зі специфікації → реалізація →
> перевірка DoD → деплой (з Е-1 кожен цикл закінчується на живому проді) →
> фіксація метрик. Якщо обсяг циклу не закривається — ріжемо функціонал
> (BC-GOAL-01), а не розтягуємо цикл.

---

## 1. Цикли

### Е-1 · Каркас і деплой

**Обсяг:** Nx-монорепо (`apps/web`, `apps/api`, `libs/*`); NestJS + Prisma + MySQL
(порожня схема + перша міграція); Angular SPA «hello»; вибір UI-набору (делеговано
агенту, ADR-0001); Dockerfile (multi-stage, ADR-0002); docker-compose для local;
health-ендпоінт; сервіс на Railway (MySQL + Volume заведені, ADR-0005);
**інструментарій документації** — адаптація з репозиторію `service-desk-prd` під
структуру `docs/` і спрощену методологію (без журналу `Р-`): детермінований лінтер
`tools/check-docs.py` (цілісність посилань, версійні крос-посилання, коди
FR-/NFR-/TC-/BC- означені в `docs/PRD.md`, реєстр `docs/adr/` ↔ README) +
pre-commit hook; спрощений скіл `/record-decision` (рішення → ADR або правка PRD
+ журнал змін + прогін лінтера).
**DoD:** прод-URL відкриває SPA; `/api/health` зелений; локальний старт — одна команда;
у підсумках циклу зафіксовано обраний UI-набір; лінтер документації проходить чисто
й запускається pre-commit-хуком.

### Е-2 · Автентифікація OTP

**Обсяг:** FR-AUTH-01…04; NFR-SEC-01/02; таблиця OTP + rate-limit (ADR-0004);
інтеграція TurboSMS + dev-фолбек; сесії (httpOnly cookie), guard на API; екран
входу/реєстрації; профіль (ім'я) — мінімально.
**DoD:** на проді реальний номер отримує SMS і входить; у local код видно без SMS;
ліміти й TTL перевірені тестами.

### Е-3 · Будинки + створення заявки

**Обсяг:** FR-HOUSE-01/02; FR-TICKET-01…04; FR-ACCESS-01 (наскрізно з цього циклу);
схема БД §9 (user/house/ticket); екран довідника будинків; форма створення/редагування
заявки; картка заявки (без стрічки).
**DoD:** на проді можна завести будинок і створити/відредагувати заявку; чужі
об'єкти недоступні (404-стиль, перевірено тестом).

### Е-4 · Життєвий цикл і стрічка

**Обсяг:** FR-STATUS-01…03; FR-DUE-01; FR-FEED-01/02; переходи статусів у картці
(лише дозволені кнопки + валідація API); єдина стрічка: записи + системні події.
**DoD:** повний прохід Нова → В роботі → Виконана → Закрита (+ повторне відкриття,
відхилення) на проді; всі зміни видно в стрічці; заборонені переходи відбиваються API.

### Е-5 · Список, фільтри, прострочення

**Обсяг:** FR-LIST-01…04; FR-DUE-02; NFR-PERF-01; список із фільтрами, пошуком,
сортуванням, пагінацією; підсвічування прострочених у списку й картці; мобільна
верстка списку (NFR-COMPAT-01).
**DoD:** усі фільтри комбінуються; прострочена активна заявка підсвічена, завершена —
ні; список зручний зі смартфона.

### Е-6 · Фото-вкладення

**Обсяг:** FR-ATTACH-01…03; NFR-STOR-01 (ADR-0003); аплоад із валідацією (тип/розмір/
кількість), мініатюри й перегляд, видалення з системною подією; роздача через API
з перевіркою власника; рішення щодо HEIC (ризик §11 PRD — за потреби звузити типи
правкою FR-ATTACH-01).
**DoD:** фото зі смартфона прикріплюється, переглядається, видаляється на проді;
файл переживає редеплой (volume); прямий URL чужого файла → 404.

### Е-7 · Поліровка, наскрізна перевірка, ретроспектива POC

**Обсяг:** прохід усіх сценаріїв §6 на проді; звірка кожного FR/NFR зі
специфікацією (чеклист); дрібні UX-правки; NFR-OBS-01 (логи без PII); **звіт POC**:
метрики всіх циклів (§2), оцінка якості коду й UI, висновки для основного проєкту.
**DoD:** чеклист FR/NFR закритий; звіт POC написано; продуктовий критерій PRD §3.2
досягнуто.

## 2. Метрики кожного циклу

Фіксуються одразу після циклу (у підсумках циклу — файл `docs/cycles/E-N.md`
цього репозиторію або аналог):

| Метрика | Як міряємо |
|---|---|
| Час циклу | Календарний час + чистий час людини (рев'ю, правки) |
| Вартість | Токени/кошти агентів + витрати Railway/TurboSMS за період |
| Ітерації | Кількість повторних заходів агента до досягнення DoD |
| Дефекти | Знайдені після завершення циклу (у наступних циклах) |
| Дотримання специфікації | Відхилення від FR/NFR/ADR, самовільні рішення агента |

## 3. Правила скоупу

- Порядок циклів фіксований — кожен наступний спирається на попередній.
- **Якщо йдемо швидше:** дозволені злиття Е-3+Е-4 та Е-5+Е-6 (мінімум — 5 циклів).
- **Якщо цикл не вміщується:** ріжемо обсяг (наприклад, Е-5 без пошуку → пошук
  переїжджає в Е-7), фіксуючи це правкою PRD + журнал змін. Загальна межа —
  7 циклів (BC-GOAL-01).
- Інженерні рішення під час розробки — новим ADR у [`adr/`](./adr/README.md);
  продуктові — правкою [PRD](./PRD.md) (журнал `Р-` не ведеться).
