# MVP Capability Plan — Сервіс-деск Mini (POC)

> **Версія:** 1.6 · **Створено:** 2026-07-07 · **Оновлено:** 2026-07-08
> **Спирається на:** [PRD](./PRD.md) (v1.2) · [agent-plan](./agent-plan.md) (v1.4, рамка циклів) · [журнал](./assumptions-open-questions.md) (Р-11)
>
> **Робочий план** агентної розробки (Р-11): одиниця роботи — **capability-зріз**.
> Згенеровано скілом `/slice-plan` з PRD v1.2; цикли Е-1…Е-7 з agent-plan лишаються
> рамкою метрик і порядку — кожен зріз мапиться на цикл, порядок циклів збережено.
>
> **Правила нарізки:** один зріз = одна capability = одна одиниця agentic-роботи =
> один openspec change = один real-behavior-proof. Постачання — **trunk-based**
> (ADR-0008, трунк — гілка `service-desk-mini`, ADR-0012): зріз лягає в трунк
> звʼязною серією комітів `feat(S-NN): …`, без робочих гілок і PR; єдиний PR —
> фінальна здача (`service-desk-mini` → `main` форку). Кожен зріз
> **вертикальний** (UI → API → БД, повний контур) і **self-contained** — застосунок
> придатний до прод-деплою одразу після приземлення зрізу в main. Що не
> вміщується — ріжеться (BC-GOAL-01). Покриття вимог — у
> [traceability-matrix](./traceability-matrix.md).

---

## 1. Зрізи

### S-01 · Хребет застосунку і прод-деплой — Е-1

- **Мета:** «ходячий скелет»: порожній, але живий застосунок на проді з БД і health-чеком.
- **FR/NFR:** TC-STACK-01/02 · TC-OPS-01 · NFR-SEC-04 · NFR-REL-01 (бекапи) · NFR-OBS-01 (health-частина).
- **Вертикальний контур:** UI — hello-сторінка на обраному UI-наборі (закриває В-01), роздається NestJS-ом; API — `/api/health`; БД — Prisma + MySQL, порожня схема, перша міграція. Інфраструктура: Dockerfile (multi-stage), docker-compose (local MySQL), сервіс на Railway (MySQL + Volume).
- **Acceptance-сценарії:**
  - Given чистий клон репо, When розробник виконує одну команду локального старту, Then SPA відкривається на 4200/3000 і `/api/health` зелений.
  - Given прод-URL Railway, When відкрити його в браузері, Then SPA рендериться, `/api/health` повертає ok, зʼєднання з MySQL живе.
- **Залежності:** —
- **Non-goals:** доменна модель, автентифікація, будь-який продуктовий функціонал.

### S-02 · Автентифікація OTP — Е-2

- **Мета:** користувач реєструється/входить за телефоном через SMS-код і має тривалу сесію.
- **FR/NFR:** FR-AUTH-01…04 · NFR-SEC-01 · NFR-SEC-02.
- **Вертикальний контур:** UI — екран входу (телефон → код), дія «Вийти», мінімальний профіль (імʼя); API — request-otp / verify-otp / logout, guard на всі захищені ендпоінти, сесія в httpOnly cookie; БД — `user`, `otp_code` (хеші кодів, TTL, спроби, rate-limit лічильники); TurboSMS + dev-фолбек (код у лог, ADR-0004).
- **Acceptance-сценарії:**
  - Given новий номер +380…, When користувач запитує код і вводить його з SMS (у local — з логу), Then створюється акаунт і користувач у застосунку.
  - Given 6-й запит SMS на номер за добу або 2-й за 60 с, When користувач запитує код, Then зрозуміла помилка rate-limit (FR-AUTH-03), SMS не йде.
  - Given введено 5 неправильних кодів, When вводиться 6-й, Then код анульовано, потрібен новий (FR-AUTH-02).
  - Given успішний вхід, When користувач повертається наступного дня, Then сесія жива без повторного OTP (FR-AUTH-04).
- **Залежності:** S-01.
- **Non-goals:** будь-які доменні дані; ролі; відновлення доступу поза OTP.

### S-03 · Довідник будинків — Е-3

- **Мета:** користувач веде власний довідник будинків, ізольований від чужих очей.
- **FR/NFR:** FR-HOUSE-01/02 · FR-ACCESS-01 · NFR-SEC-03 (запроваджуються тут, далі — наскрізно в кожному зрізі).
- **Вертикальний контур:** UI — екран довідника (список, створення, редагування) + редирект авторизованого користувача з `/login` на головну (low-знахідка ревʼю S-02, рішення 2026-07-08); API — CRUD будинків з перевіркою власника, 404-стиль для чужого/неіснуючого; БД — `house`.
- **Acceptance-сценарії:**
  - Given залогінений користувач, When він створює будинок «Шевченка 12» і редагує примітку, Then будинок видно у його довіднику з оновленими даними.
  - Given будинок користувача А, When користувач Б відкриває його за прямим URL, Then відповідь 404-стилю (FR-ACCESS-01).
  - Given будинок із заявкою (з S-04+), When користувач видаляє його, Then зрозуміла помилка; будинок без заявок — видаляється (FR-HOUSE-02).
- **Залежності:** S-02.
- **Non-goals:** заявки; імпорт/синхронізація довідника.

### S-04 · Заявка: створення, редагування, картка — Е-3

- **Мета:** користувач створює заявку з усіма атрибутами і бачить її картку.
- **FR/NFR:** FR-TICKET-01/02/04 _(FR-TICKET-03 — у S-05, бо системні події потребують стрічки)_.
- **Вертикальний контур:** UI — форма створення/редагування, картка заявки (без стрічки), звідси й далі — мобільний-пріоритет; API — create/update/get з перевіркою власника; БД — `ticket` (enum-и категорій/пріоритетів/статусів §5.2–5.3, номер #N = BIGINT AI).
- **Acceptance-сценарії:**
  - Given будинок у довіднику, When користувач створює заявку з назвою, будинком і категорією, Then заявка у статусі `Нова` з наскрізним номером #N видна в картці (FR-TICKET-01/02, FR-STATUS-01-частково).
  - Given створена заявка, When користувач редагує виконавця й термін, Then зміни збережені й видні в картці (FR-TICKET-01, FR-DUE-01-частково).
  - Given заявка користувача А, When користувач Б відкриває її картку за URL, Then 404-стиль.
- **Залежності:** S-03.
- **Non-goals:** переходи статусів; стрічка; системні події; вкладення; список.

### S-05 · Життєвий цикл і стрічка — Е-4

- **Мета:** заявка проходить повний 5-статусний цикл, і вся історія видна в єдиній стрічці.
- **FR/NFR:** FR-STATUS-01…03 · FR-FEED-01/02 · FR-TICKET-03 · FR-DUE-01.
- **Вертикальний контур:** UI — кнопки лише дозволених переходів у картці, стрічка (записи + системні події, візуально різні), поле запису; API — переходи з валідацією таблиці §5.1 (заборонений → помилка), append-only записи; БД — `ticket_feed_item`.
- **Acceptance-сценарії:**
  - Given заявка `Нова`, When користувач проводить її Нова → В роботі → Виконана → Закрита, Then кожен перехід зафіксовано системною подією у стрічці (хто/коли/звідки/куди).
  - Given заявка `Закрита`, When API отримує запит переходу в `В роботі`, Then помилка — перехід заборонено (FR-STATUS-02).
  - Given заявка `Виконана`, When проблема повернулась і користувач повторно відкриває, Then статус `В роботі`, подія у стрічці.
  - Given зміна виконавця/терміну/будинку/категорії/пріоритету, When користувач зберігає, Then системна подія у стрічці (FR-TICKET-03).
- **Залежності:** S-04.
- **Non-goals:** сповіщення; автопереходи; SLA; редагування/видалення записів.

### S-06 · Список, фільтри, пошук, прострочення — Е-5

- **Мета:** користувач знаходить будь-яку свою заявку і одразу бачить прострочені.
- **FR/NFR:** FR-LIST-01…04 · FR-DUE-02 · NFR-PERF-01 · NFR-COMPAT-01.
- **Вертикальний контур:** UI — екран списку з фільтрами (статус/пресет «активні», будинок, категорія, пріоритет), LIKE-пошук, сортування, пагінація/нескінченний скрол, підсвічування прострочених у списку та картці, мобільна верстка; API — list-ендпоінт з комбінованими фільтрами, пошуком, сортуванням, пагінацією; БД — індекси під фільтри (`user_id+status`, `user_id+house_id`).
- **Acceptance-сценарії:**
  - Given заявки різних статусів/будинків, When користувач комбінує фільтри «активні» + будинок, Then список показує лише відповідні заявки (FR-LIST-02).
  - Given заявка з терміном учора у статусі `В роботі` та інша `Закрита` з тим самим терміном, When відкрито список, Then перша підсвічена як прострочена, друга — ні (FR-DUE-02).
  - Given пошуковий запит за прізвищем заявника, When користувач шукає, Then знаходяться заявки за назвою/описом/заявником/виконавцем (FR-LIST-03).
- **Залежності:** S-05.
- **Non-goals:** повнотекстовий пошук; збережені фільтри; експорт.

### S-07 · Фото-вкладення — Е-6

- **Мета:** користувач прикріплює фото до заявки, переглядає і видаляє їх; файли живуть на Volume і недоступні чужим.
- **FR/NFR:** FR-ATTACH-01…03 · NFR-STOR-01. Тут закривається В-02 (HEIC): рішення фіксується Р-записом (+ правка FR-ATTACH-01 за потреби).
- **Вертикальний контур:** UI — аплоад у картці, мініатюри, повний розмір, видалення; API — upload з валідацією (тип/розмір/кількість), роздача лише через API з перевіркою власника (404-стиль), видалення з системною подією; БД/диск — `attachment` + файли на Railway Volume (шлях з env).
- **Acceptance-сценарії:**
  - Given заявка, When користувач додає фото 3 МБ JPEG зі смартфона, Then мініатюра в картці, повний розмір відкривається, у стрічці — системна подія.
  - Given файл 12 МБ або PDF, When користувач пробує додати, Then зрозуміла помилка валідації (FR-ATTACH-01).
  - Given вкладення користувача А, When користувач Б відкриває його прямий URL, Then 404-стиль (FR-ATTACH-03).
  - Given редеплой застосунку, When користувач відкриває картку після редеплою, Then фото на місці (NFR-STOR-01).
- **Залежності:** S-05 (системні події у стрічці).
- **Non-goals:** обробка зображень (ресайз/EXIF); не-зображення; S3.

### S-08 · Поліровка, наскрізна перевірка, ретроспектива POC — Е-7

- **Мета:** всі сценарії §6 проходять на проді; POC закритий звітом.
- **FR/NFR:** NFR-OBS-01 (повністю: структуровані логи без PII) · звірка всіх FR/NFR (traceability-matrix заповнена на 100%) · продуктовий критерій PRD §3.2.
- **Вертикальний контур:** наскрізний прохід S1–S7 на проді; дрібні UX-правки; звіт POC (метрики циклів, якість, висновки для основного проєкту).
- **Acceptance-сценарії:**
  - Given реальний користувач і прод, When він проходить happy-path §6 (реєстрація → будинок → заявка з фото → цикл до `Закрита` → пошук у списку), Then усе працює без ручних втручань.
  - Given traceability-matrix, When звіряється кожен FR/NFR, Then кожен має spec, тест і demo check.
- **Залежності:** S-01…S-07.
- **Non-goals:** новий функціонал.

## 2. Definition of Done кожного зрізу

Зріз вважається готовим, коли:

1. `proposal.md`, `design.md`, `tasks.md`, spec-дельти — заповнені й валідовані; всі чекбокси у `tasks.md` — `[x]`.
2. `npm run verify` проходить (format, lint, typecheck, fallow audit, openspec validate, tests, build — все блокуюче).
3. Smoke-тест на реальній БД: створили / змінили / видалили дані зрізу, перевірили інваріанти.
4. E2E-сценарії критичних шляхів зрізу працюють у Playwright (`web-e2e`) — виводяться з acceptance-сценаріїв.
5. Adversarial-ревʼю субагентом `slice-reviewer` (ADR-0010): чистий контекст, інша модель, один прогін по diff зрізу. `critical`/`high` знахідки — виправлені (з повторним verify); `medium`/`low` — на розсуд автора, диспозиції фіксуються в ретро.
6. Launch-and-look: застосунок запущено, happy-path зрізу пройдено очима, працює (без записів — факт перевірки нотується в current-state).
7. `npx openspec validate <change> --strict` — pass; change заархівовано; `npx openspec list` порожній перед оголошенням зрізу готовим.
8. `docs/current-state.md` оновлено: phase, done, next 1–2 tasks, blockers.
9. [`docs/traceability-matrix.md`](./traceability-matrix.md) оновлено: FR → spec → тест → demo check.
10. Ретроспектива сесії скілом `/slice-retro`: метрики й тертя → [`docs/cycles/S-NN.md`](./cycles/README.md); дрібні процесні покращення (≤3) застосовано, нормативні — запропоновано користувачу.

## 3. Процесний трек (поза зрізами)

Роботи з agent-plan Е-1, що не є capability-зрізами (виконуються окремими PR
без openspec change):

- ✅ Quality gates: `npm run verify` + хуки Claude Code (fallow, ADR-0006).
- ✅ OpenSpec як робочий шар SDD (ADR-0007) + скіл `/slice-plan`.
- ☐ Інструментарій документації: лінтер `tools/check-docs.py` + скіл `/record-decision` (обсяг Е-1 agent-plan v1.1; pre-commit-частину вже покриває verify gate).
- ☐ Design-check у `verify` (закриває В-04): статична перевірка, що UI-код використовує токени теми Material (`var(--mat-sys-*)`) замість хардкоду кольорів/розмірів — stylelint-правило або скрипт. Виконати **перед стартом S-03** (перший справжній UI; рішення ретро S-01, 2026-07-08).
- ✅ Флоу постачання: trunk-based, без робочих PR; один фінальний PR здачі (ADR-0008 — окремий PR-скіл не потрібен).

## 4. Аудит agent-plan (знахідки прогону `/slice-plan`)

1. **Е-3 містить дві capability** (будинки і заявки) — розрізано на S-03 та S-04: кожен окремо deployable і demo-able. Порядок циклів не порушено.
2. **FR-TICKET-03 перенесено з Е-3 до S-05 (Е-4):** вимога фіксує зміни полів системними подіями, а стрічка зʼявляється лише в Е-4 — у Е-3 вона була б нетестованою.
3. **Обсяг Е-1 виконано частково:** скафолд є; деплой, Prisma/MySQL, Dockerfile, health — у S-01; інструментарій документації винесено у процесний трек (§3).
4. FR-ACCESS-01/NFR-SEC-03 — наскрізні: первинне покриття в S-03, кожен наступний зріз додає перевірку ізоляції для своїх обʼєктів (закріплено в traceability-matrix).
