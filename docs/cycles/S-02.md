## Що спрацювало

- **Fail-fast з ADR-0004 окупився наживо**: перший прод-деплой упав саме там,
  де спроєктовано — `SMS_MODE=turbosms requires TURBOSMS_TOKEN`, Railway
  лишив попередній деплоймент живим. Конфіг-помилка не долетіла до юзерів.
- Adversarial-ревʼю (ADR-0010) знову дало реальні знахідки: 2 high — обидві
  конкурентні гонки (TOCTOU rate-limit, неатомарний лічильник спроб), які
  жоден існуючий тест не ловив. Виправлення: per-phone `KeyedMutex` + atomic
  increment; додано конкурентні регресії в unit та api-e2e.
- Заморожений діапазон ревʼю кінцевим SHA (рішення ретро S-01) — жодних
  «діапазон поїхав»; ревʼюер підтвердив обидва SHA.
- `devCode` у dev-відповіді (ADR-0004 дозволяє) зробив api-e2e та Playwright
  детермінованими без скрейпінгу логів; рейт-ліміти в тестах обійдено
  унікальними номерами й прямими SQL-фікстурами — без жодного `sleep`.
- fallow зловив справжні проблеми ще до ревʼю: незадекларований
  `@angular/forms` (працював лише через транзитивний peer), циклічний імпорт
  interceptor → facade → api, 3 функції над порогом складності.
- Ревʼюер пережив обрив API-зʼєднання: `SendMessage` відновив агента з
  транскрипту, пас не перезапускався з нуля.

## Тертя і знахідки

- (середовище) **Railway staged-змінні**: додані через дашборд змінні не
  застосовуються до натискання «Apply changes/Deploy» — CLI їх не бачить,
  деплой падає. Коштувало один цикл деплою і ~15 хв зʼясувань (скриншот
  користувача). CLI-шлях `railway variables --set` застосовує одразу.
- (інструменти) **Nx target lock**: фоновий `nx serve api` заблокував
  `nx e2e api-e2e` (чекає `api:serve` в іншому процесі) — перший запуск e2e
  упав незрозумілою помилкою globalSetup. Фонові serve треба зупиняти перед
  e2e.
- (інструменти) fallow false positive: `unused-class-member` для
  `AuthFacade.sessionLost`, який викликається через `inject()`-типізований
  інстанс в інтерсепторі — тип-залежні виклики не трасуються; інлайн
  suppression із коментарем.
- (інструменти) fallow `ignorePatterns` `**/src/support/**` (для jest-інфри)
  сховав імпорт `mariadb` у новому db-хелпері api-e2e → «unused dependency»;
  вирішено переносом файла до `api-e2e/src/api/`, конфіг не розширювали.
- (інструменти) `railway logs` без аргументів показує останній *успішний*
  деплоймент — логи впалого шукати через `railway logs <deployment-id>`
  (повний ID з `railway deployment list --json`).
- (документація vs практика) PRD §9.1 не містить сутності `session`, хоча
  server-side сесія зафіксована ADR-0004; також «лічильники добового ліміту»
  в `otp_code` реалізовано похідними запитами, а не колонками (design D1) —
  формулювання PRD допускає прочитання «стовпці».
