# ADR-0001 — Успадкування стеку основного проєкту: Nx · Angular · NestJS · Prisma · MySQL

- **Статус:** Accepted
- **Дата:** 2026-07-06
- **Замінює / Замінено:** —
- **Трасування:** TC-STACK-01 · BC-GOAL-01 · кореневі ADR-0002/0003/0004/0005 (основний проєкт)

## Контекст і проблема

POC має два призначення: робочий міні-застосунок і, головне, **оцінка агентної
розробки на стеку основного проєкту**. Якщо взяти «зручніший для POC» стек
(наприклад, Next.js + SQLite), висновки про продуктивність агентів не переносяться
на основний проєкт.

## Драйвери рішення

- **BC-GOAL-01** — показовість POC для основного проєкту важливіша за локальну
  оптимальність вибору.
- Замовник явно зафіксував: Angular, Nx, Railway, контейнеризація — лишаються.

## Розглянуті варіанти

- **A — Успадкувати стек основного проєкту** (Nx-монорепо; Angular SPA; NestJS +
  Prisma + Prisma Migrate; REST/JSON + OpenAPI; MySQL).
- **B — Полегшений стек під POC** (інший фреймворк / вбудована БД).

## Рішення

Обрано **A**. Конкретно:

- **Nx-монорепо**: `apps/web` (Angular SPA), `apps/api` (NestJS), `libs/*`
  (спільні типи/DTO, за потреби — UI-бібліотека). Примусові межі модулів Nx.
- **Бекенд:** NestJS + **Prisma** (єдина БД — власна, тож сирий SQL/Kysely
  з кореневого ADR-0002 тут не потрібен). Міграції — **Prisma Migrate**.
- **API:** REST/JSON з **OpenAPI**, генерованим із коду (Nest Swagger).
- **БД:** **MySQL** — як в основному проєкті (порівнянність схем і досвіду),
  Railway надає керовану MySQL.

Вибір компонентного набору UI (Material vs PrimeNG) в основному проєкті відкритий
(кореневий ADR-0006); для POC він **делегується агентній розробці** в межах циклу
каркаса — обраний набір фіксується в підсумках циклу і є одним із результатів POC.

## Наслідки

- **Позитивні:** висновки POC (швидкість, якість, вартість агентних циклів)
  прямо переносяться на основний проєкт; готові референсні рішення зі стеку.
- **Ціна:** стек важчий за мінімально можливий для такого застосунку — свідома
  плата за показовість.
