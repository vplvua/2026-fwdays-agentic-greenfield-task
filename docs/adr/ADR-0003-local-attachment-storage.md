# ADR-0003 — Вкладення на локальному диску (Railway Volume), роздача через API

- **Статус:** Accepted
- **Дата:** 2026-07-06
- **Замінює / Замінено:** —
- **Трасування:** TC-MEDIA-01 · NFR-STOR-01 · NFR-REL-01 · FR-ATTACH-01…03

## Контекст і проблема

Основний проєкт зберігає фото у зовнішньому S3-сумісному сховищі — ефемерна ФС
контейнерів Railway робить це обов'язковим (кореневе рішення Р-89). Для POC
замовник явно обрав **локальне зберігання на сервері**. Треба розв'язати конфлікт
з ефемерною ФС і зафіксувати межі такого рішення.

## Драйвери рішення

- Явна вимога замовника: без S3, файли локально.
- **BC-COST-01** — без зовнішнього сховища (менше сервісів, ключів, коштів).
- Ефемерна ФС Railway: файли поза volume зникають при кожному деплої.

## Розглянуті варіанти

- **A — Railway Volume:** персистентний том, змонтований у сервіс; файли живуть
  між деплоями.
- **B — Просто локальна тека** без volume (файли втрачаються при деплої) —
  непрацездатно навіть для POC.
- **C — S3-сумісне сховище** — суперечить явній вимозі; зайвий сервіс для POC.

## Рішення

Обрано **A — Railway Volume**:

- шлях монтування задається env-змінною (локально — звичайна тека);
- зберігаємо **оригінал** файла (без ресайзу/обробки — поза скоупом), ім'я на
  диску — згенерований ідентифікатор, оригінальне ім'я — в БД;
- файли **роздаються лише через API** з перевіркою власника (FR-ACCESS-01);
  публічної статичної роздачі теки немає;
- ліміти FR-ATTACH-01 (тип, ≤10 МБ, ≤10 файлів) валідуються на бекенді.

## Наслідки

- **Позитивні:** нуль зовнішніх залежностей для файлів; тривіальна реалізація
  (fs + stream); дешево.
- **Ціна / прийняті ризики:** volume не бекапиться як БД — **втрата фото при
  втраті volume прийнята** (NFR-REL-01); volume прив'язує сервіс до **одного
  інстанса** (TC-MEDIA-01) — горизонтального масштабування немає.
- **Шлях еволюції:** інтерфейс сховища у коді ізолювати (локальний драйвер зараз,
  S3-драйвер — якщо POC переросте у продукт), як у кореневому ADR-0010.
