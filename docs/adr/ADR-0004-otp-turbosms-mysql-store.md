# ADR-0004 — OTP: SMS через TurboSMS + dev-фолбек; транзитний стор — таблиця MySQL

- **Статус:** Accepted
- **Дата:** 2026-07-06
- **Замінює / Замінено:** —
- **Трасування:** TC-AUTH-01 · FR-AUTH-01…04 · NFR-SEC-01/02 · BC-COST-01 · кореневі ADR-0007, Р-90

## Контекст і проблема

Реєстрація/вхід — OTP за номером телефону (FR-AUTH-01). Треба обрати канал
доставки, спосіб зберігання транзитних даних (код, лічильники, rate-limit) і не
спалювати SMS-бюджет під час агентної розробки й тестів.

## Драйвери рішення

- Порівнянність з основним проєктом (там: SMS-only через TurboSMS, стор — таблиця
  MySQL; кореневі Р-90 / ADR-0007).
- **BC-COST-01** — SMS коштують грошей; агентні цикли ганяють автентифікацію часто.
- NFR-SEC-01/02 — TTL, ліміт спроб, rate-limit, зберігання лише хешів кодів.

## Розглянуті варіанти

- **Канал:** TurboSMS (як в основному) / інший шлюз / e-mail-OTP.
- **Стор:** таблиця MySQL / Redis.
- **Дев-режим:** без окремого режиму (реальні SMS завжди) / фолбек без надсилання.

## Рішення

- **Канал — TurboSMS** (той самий кабінет/досвід, що й основний проєкт).
  Інтеграція ізолюється інтерфейсом `SmsSender`.
- **Dev-фолбек обов'язковий:** поза продом (env-перемикач) SMS не надсилається —
  код пишеться у лог/відповідь дев-режиму; e2e-тести працюють без шлюзу й витрат.
- **Транзитний стор — таблиця MySQL** (як кореневий ADR-0007): TTL-поле, лічильники
  спроб і добових лімітів, очистка простроченого фоновою задачею. Redis не вводимо.
- **Сесія** після входу — server-side токен у httpOnly cookie (NFR-SEC-01),
  тривалість за FR-AUTH-04.

## Наслідки

- **Позитивні:** нуль SMS-витрат у розробці й CI; жодного додаткового
  інфра-компонента; пряме перевикористання підходу основного проєкту.
- **Ціна:** dev-фолбек — це гілка поведінки, яку треба явно закрити на проді
  (перевірка env на старті: прод без TurboSMS-ключа не стартує).
