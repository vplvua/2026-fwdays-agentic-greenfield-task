# ADR-0007 — OpenSpec як робочий шар SDD: ієрархія правди та цикл слайсу

- **Статус:** Accepted
- **Дата:** 2026-07-07
- **Замінює / Замінено:** —
- **Трасування:** BC-GOAL-01 · BC-PRIN-01 · Р-01 · [agent-plan](../agent-plan.md) · ADR-0006

## Контекст і проблема

Нормативний контур POC уже існує: PRD з кодами `FR-*/NFR-*/TC-*/BC-*`, ADR,
глосарій, журнал `Р-/П-/В-`. Але агентним циклам бракує механізованого робочого
шару: формалізованих proposal → tasks → apply → archive з валідацією, який
структурує одиницю агентної роботи та її життєвий цикл. OpenSpec дає саме це,
проте приносить власну структуру специфікацій — без чітких правил виникло б два
конкурентні джерела правди.

## Драйвери рішення

- Механізований цикл зміни (proposal → apply → archive) з CLI-валідацією.
- Інтеграція з Claude Code: слеш-команди `/opsx:*` і скіли з коробки.
- Збереження нормативності PRD і наявних процесів (`Р-` журнал, ADR).

## Розглянуті варіанти

- **A — OpenSpec** (`@fission-ai/openspec`, схема spec-driven).
- **B — власні markdown-конвенції** без інструменту: без валідації і archive.
- **C — специфікації лише в PRD:** нема одиниці роботи для агентного циклу.

## Рішення

Обрано **A** з такими правилами:

- **Ієрархія правди:** `docs/PRD.md` — нормативний; `openspec/specs/` —
  похідний робочий шар для агентів; кожна вимога специфікації трасується на
  FR-коди; конфлікт вирішується на користь PRD. Продуктові рішення не
  приймаються в openspec-артефактах — лише через `Р-` запис + правку PRD;
  інженерні — через ADR.
- **Одиниця роботи:** один change = один capability-зріз із
  `docs/mvp-capability-plan.md` = один PR. Зріз вертикальний (UI → API → БД)
  і self-contained — придатний до продакшн-деплою одразу після мержу.
- **Цикл зрізу:** `/opsx:propose` з посиланням на пункт плану → apply
  (імплементація tasks) → `npm run verify` → браузерна валідація
  acceptance-сценаріїв зрізу через Playwright (`web-e2e`) → `/opsx:archive` →
  `npx openspec list` порожній → PR.
- **Валідація в gate:** `openspec validate --all --strict` включено до
  `npm run verify` (на порожньому проєкті — no-op, тож фактично діє з
  першого change).
- **Мова:** openspec-артефакти — англійською (внутрішні робочі документи);
  правила й контекст — в `openspec/config.yaml`.

## Наслідки

- **Позитивні:** формалізована одиниця агентної роботи з валідованим життєвим
  циклом; правила зашиті в `config.yaml` і читаються агентом при кожному
  артефакті; PRD лишається єдиним джерелом вимог.
- **Ціна:** дублювання вимог PRD ↔ specs потребує дисципліни трасування;
  ще одна devDependency і набір слеш-команд; браузерна валідація перед
  archive подовжує цикл (прийнято — це і є maker ≠ checker).
