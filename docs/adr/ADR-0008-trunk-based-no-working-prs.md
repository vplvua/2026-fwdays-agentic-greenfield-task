# ADR-0008 — Спрощений флоу постачання: коміти в main, один фінальний PR здачі

- **Статус:** Accepted
- **Дата:** 2026-07-07
- **Замінює / Замінено:** змінює ADR-0007 у частині PR-флоу
- **Трасування:** BC-PRIN-01 · BC-GOAL-01 · Р-11

## Контекст і проблема

ADR-0007 закріпив «один capability-зріз = один PR». Але цей репозиторій — форк
репозиторію курсу, здача — **один фінальний PR** в upstream за шаблоном
`.github/pull_request_template.md`. Робочі PR зрізів усередині форку не мають
адресата-ревʼюера і додають церемонію (гілки, merge), що суперечить BC-PRIN-01.

## Рішення

- Розробка — **trunk-based**: зрізи комітяться прямо в `main`, без робочих
  гілок і PR.
- Межа зрізу в історії — коміти з конвенцією **`feat(S-NN): …`** (процесні
  роботи — `chore(E-N)`/`chore:`); завершення зрізу фіксується оновленням
  `docs/current-state.md` і `docs/traceability-matrix.md` (DoD п.7–8).
- Формула ADR-0007 уточнюється: один зріз = один openspec change = **одна
  зв'язна серія комітів у main** = один real-behavior proof. Решта ADR-0007
  (ієрархія правди, цикл зрізу, DoD, validate у gate) — без змін.
- **Єдиний PR** — фінальна здача домашки в upstream за шаблоном
  `.github/pull_request_template.md` (справжнє імʼя, відео-демо 1–2 хв, опис
  практик agentic engineering).

## Наслідки

- **Позитивні:** мінімум церемонії; кожен коміт у main уже пройшов verify gate
  (pre-commit хук); історія читається по зрізах.
- **Ціна:** немає PR-ревʼю CodeRabbit по ходу роботи — чекером лишаються
  verify gate, fallow, Playwright і launch-and-look; CodeRabbit побачить усе
  разом у фінальному PR.
