# ADR-0002 — Один сервіс, один контейнер: NestJS роздає статику SPA

- **Статус:** Accepted
- **Дата:** 2026-07-06
- **Замінює / Замінено:** —
- **Трасування:** TC-STACK-02 · NFR-SEC-01 · TC-MEDIA-01 (один інстанс) · BC-COST-01

## Контекст і проблема

Основний проєкт розгортає SPA та API за спільним reverse-proxy (один origin).
Для POC треба вирішити: відтворювати цю топологію (2–3 сервіси на Railway) чи
спростити до одного сервіса.

## Драйвери рішення

- **BC-COST-01** — мінімальна вартість Railway і мінімум інфраструктурних рухів
  на агентний цикл.
- **NFR-SEC-01** — сесійна cookie вимагає одного origin (без CORS/CSRF-ускладнень).
- TC-MEDIA-01 і так обмежує застосунок одним інстансом (локальний диск).

## Розглянуті варіанти

- **A — Один контейнер:** NestJS віддає `/api/*` та статику збірки Angular
  (SPA-fallback на `index.html`).
- **B — Як в основному проєкті:** окремий reverse-proxy + окремі фронт- і
  бек-сервіси.

## Рішення

Обрано **A**. Один Docker-образ: multi-stage збірка (Nx білдить SPA і API →
рантайм-образ з API, що роздає статику через ServeStatic). Один origin, один
сервіс Railway, один деплой.

## Наслідки

- **Позитивні:** один деплой-юніт (простіші агентні цикли: «зібрав образ —
  задеплоїв»); один origin без CORS; дешевше на Railway.
- **Ціна:** топологія відрізняється від основного проєкту (там reverse-proxy) —
  прийнятно, бо на висновки POC щодо розробки це майже не впливає.
- **Обмеження:** масштабування — лише вертикальне (узгоджено з TC-MEDIA-01).
