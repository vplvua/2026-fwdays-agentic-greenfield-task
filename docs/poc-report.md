# Звіт POC — Сервіс-деск Mini

> **Версія:** 1.2 (фінальна) · **Дата:** 2026-07-09
> Підсумковий звіт POC за PRD [§3.1/§3.3](./PRD.md): метрики агентних циклів,
> оцінка якості, висновки для основного проєкту. Джерела: ретроспективи
> [docs/cycles/](./cycles/README.md), [traceability-matrix](./traceability-matrix.md),
> [журнал рішень](./assumptions-open-questions.md).

## 1. Резюме

POC відбувся: **8 capability-зрізів у 7 агентних циклах** (Е-1…Е-7,
BC-GOAL-01 дотримано), робочий застосунок на проді (Railway) з реальним
SMS-входом, усі FR §7 закриті кодом, тестами і demo-чеками. Ключова цифра
для рішення про основний проєкт: **$528.94 і ≈8 год 17 хв чистого API-часу
на весь POC (S-01…S-08)** при **≈35 хв людського часу** — людина витрачала
час лише на продуктові рішення (Р-записи) і фізично недоступні агенту дії
(SMS-вхід, дашборд Railway, прод-прохід з реальним SMS).

Методологічний висновок у одному реченні: спека-драйв + незалежні
верифікатори (verify gate, adversarial-ревʼю, смок на реальній БД,
launch-and-look очима) зловили **всі** суттєві дефекти до проду — жодного
інциденту на проді за POC.

  `/compact` (75% витрат на ходах з контекстом >150k) і втричі важчого
  ревʼюера.

## 3. Якість результату

### 3.1. Код і тести

- Кожен зріз ішов повним циклом propose → apply → verify → смок → e2e →
  adversarial-ревʼю → launch-and-look → archive; DoD не скорочувався.
- Дефекти, що протекли між зрізами, за POC — **два в коді** (шрифт Material
  Icons не підключався: S-01 → знайдено очима в S-03; помилковий 404 на
  порожній ефективний PATCH: S-03 → знайдено смоком S-04) і **один у
  документації** («git push тригерить Railway» — S-07). Жоден не дійшов до
  користувача на проді.
- Повторюваний клас багів у згенерованому коді — **стан SPA після мутації**
  (S-04 успадкований стан форми, S-05 commit-after-reload, S-07 та сама
  рима в новому методі). Це найкращий аргумент за незалежне ревʼю: клас
  повторився навіть після того, як фікс-ідіома вже була в кодовій базі.

### 3.2. Архітектурна дисципліна

- **0 відхилень від FR/NFR/ADR у всіх семи ретро.** Мікрорішення в межах
  дизайну фіксувалися в ретро; продуктові рішення ескалювалися (Р-11…Р-13),
  інженерні — новими ADR (12 штук).
- Ідіоми переносились між зрізами без дрейфу: single-query owner-check +
  404-парність (S-03 → усі), «правила рахує сервер» (`allowedTransitions`,
  `isOverdue`), container/presentational + фасади (ADR-0009).
- PRD лишився нормативним: конфлікти spec ↔ PRD не виникали; двічі PRD
  правився керовано (Р-12 сесії, Р-13 HEIC) з бампом версії.

### 3.3. Adversarial-ревʼю (ADR-0010, maker ≠ checker)

- 7/7 ревʼю дали ≥1 реальну знахідку; сумарно **5 high** (2 падіння чистого
  клону, 2 конкурентні гонки OTP, 1 успадкований стан форми) — жодну не
  зловили тести автора.
- Ревʼюер помиляється теж: 1 знахідку (UTC у DatePipe) відхилено з доказом
  через e2e — правило «фікс ревʼюера — гіпотеза, валідувати тестами»
  зафіксовано в процесі з S-04.
- Вартість ревʼю — $1.6–5.4 за зріз (Sonnet) — непорівнянна з ціною будь-якої
  з high-знахідок на проді.

### 3.4. UI

- Згенерований Material-UI консистентний (design-check блокує хардкод
  кольорів — лише `var(--mat-sys-*)`), mobile-first перевірявся очима на
  390×844 у кожному UI-зрізі.
- Launch-and-look очима як окремий крок DoD виправдався: баг шрифту іконок
  (S-03) принципово не ловиться тестами — Playwright бачить лігатурний
  текст як валідний DOM.

## 4. Що агентна розробка за SDD робить добре (цей стек)

1. **Вертикальні зрізи за спекою — стабільно з першого проходу:** кожен зріз
   1 прохід + 1 цикл фіксів після ревʼю; 0 ревертів за POC.
2. **Незалежні верифікатори мають різні профілі влучань, і всі окупились:**
   verify gate — гігієна і складність; смок на реальній БД — те, що моки
   маскують (порожній PATCH, LIKE-екранування); Playwright — нативна
   поведінка браузера (`ngSubmit` без директиви); ревʼю — конкурентність і
   стан; очі — візуальне (шрифт іконок). Прибрати будь-який рівень — клас
   багів лишився б непокритим.
3. **Ритуал памʼяті працює:** current-state + CLAUDE.md-уроки + ретро
   переносили знання між сесіями — уроки S-03 реально економили цикли в
   S-04+ (suppression-стратегія, канонічна сюїта ізоляції, in-app
   навігація в e2e).
4. **Автономність росте з зрілістю канону:** S-01/S-02 потребували людини
   (інфраструктура, секрети), S-03…S-06 пройшли з нульовим людським часом.
5. **Гейт складності як співавтор дизайну:** обидва «спори» з fallow про
   cyclomatic закінчились табличним кодом, кращим за початковий.

## 5. Тертя і слабкі місця

1. **Вартість довгих сесій нелінійна:** S-06 коштував ×2 від сусідів через
   контекст >150k без `/compact`. Гігієна контексту — головний важіль
   економії (S-07 це підтвердив: коротша сесія — −40%).
2. **fallow — цінний, але з хибними спрацюваннями** (інпут в `@else if`,
   suppression `unused-component-input`, клони-ідіоми фасадів): ~20–30 хв
   тертя сумарно, всі обійдені документованими шляхами; кандидати на
   upstream-репорти.
3. **Оточення тестів web (vitest у браузерному емуляторі)** має граблі
   (DataTransfer, `img.loading`, TestBed-обмеження), які коштували
   червоно-зелених циклів, поки не осіли в конвенціях.
4. **Специфіка платформ пізнається лише емпірично:** Railway staged-змінні,
   `railway logs` тільки успішного деплою, відсутність GitHub-інтеграції,
   Prisma/MySQL імпліцитний FK-індекс, Railway «губить» обʼєктний `message`
   у JSON-логах (S-08). Кожне коштувало циклу діагностики і осіло в
   CLAUDE.md — ця «платіжка за перший раз» у основному проєкті вже сплачена.
5. **Дрібні продуктові рішення без ескалації** (капи полів, мітки кнопок,
   формат автора стрічки) — приймались і фіксувались у ретро; жодне не
   вистрелило, але в більшому продукті межу «мікрорішення vs Р-запис» треба
   формалізувати жорсткіше.

## 6. Висновки для основного проєкту

**Методологія (перенести як є):**

- Один зріз = один openspec change = один real-behavior-proof; DoD з
  незалежними верифікаторами (verify gate + смок реальної БД + e2e +
  adversarial-ревʼю іншою моделлю + очі) — без скорочень.
- Persistent memory bank (current-state) + інструкційний файл з уроками
  (CLAUDE.md) + ретро з ≤3 застосованими процес-фіксами за цикл — це
  механізм, завдяки якому цикли дешевшали, а не дорожчали.
- Фіксувати діапазон ревʼю кінцевим SHA; фікси ревʼюера валідувати тестами
  до прийняття.

**Методологія (додати/посилити):**

- **Гігієна контексту як норма DoD:** `/compact` (або нова сесія) між apply
  і закривальними кроками; launch-and-look виносити з основної сесії
  (Playwright MCP давав 20–26% витрат сесії).
- Бюджет-алерт на сесію (~$70 за S-06-прецедентом) як тригер зупинки й
  розбиття.
- Межу «мікрорішення в межах дизайну vs продуктове рішення» описати явним
  правилом у конфігу спек (капи/лейбли/дефолти — дизайн; усе, що видно в
  PRD-термінах — ескалація).

**Вимоги до основного проєкту (правки за досвідом POC):**

- Вимоги формулювати так, щоб нормативне SHALL-речення було першим (парсер
  openspec) і щоб UI-механіка (кшталт «стан сторінки в URL») лишалась
  дизайну, не SHALL-вимозі (урок S-06).
- В NFR обсервабіліті одразу фіксувати **формат** структурованих логів
  (allowlist полів, заборона PII) — контракт, який тестується юніт-тестами,
  а не оглядом (урок S-08).
- Класи багів для чеклиста ревʼю основного проєкту: стан SPA після мутацій,
  конкурентність (TOCTOU/атомарність), 404-парність ізоляції, порожні/крайні
  PATCH-і.
